SAP Parallel Run
The system continues to accept a manually entered SAP PO number through Phase 1 and Phase 2. The switch to internal PO number generation is a deliberate, coordinated cutover in Phase 3 — not forced. Both systems run in parallel until confidence is established.
Phase exit criteria
✓ Phase 1 Done When

All PRs carry a valid entity tag and master-driven finance codes. Vendor on every document is traceable back to the approved vendor at Price Comparison.

✓ Phase 2 Done When

A full cycle — PR → PO → GR → Billing → file posted to Bookkeeping — runs end-to-end with no manual handoff. Accounting records are complete and auditable without SAP.

✓ Phase 3 Done When

First PO is created, approved, and issued inside this system with no SAP step. SAP is no longer on the critical path for procurement.

Phase 1 — Make the System Reliable
PR Corrections & Vendor Linkage
Fix the foundation before building the accounting pipeline on top of it
4 / 9 Highest Priority
Business Objective
Every PR that enters the system today may be missing its legal entity tag or have finance codes entered as free text. We cannot build reliable Bookkeeping integration on top of that. Phase 1 fixes the data before we build on it.
Affects: Procurement, Finance Coding (Accounting), CFO
Audience Impact
Requester fills Entity Accounting uses master search Procurement locks vendor at Price Comparison CFO approval freezes vendor
Phase 1 sub-tracks
1A — PR Corrections (Foundation) 1B — PR Vendor Linkage (Enhancement)
1A — PR Corrections (must fix before anything else)
  • 1
    Entity field on PR creation
    Requester selects: NTB / NTBX / NTBI / NTBPL. Defaults to NTB. Passed to Bookkeeping in all accounting files.
    Field: dropdown, required, defaults NTB · Lock: editable until PR submitted
  • 2
    Finance Coding — master-driven
    Cost center and Expense GL sourced from Bookkeeping master via smart search. Asset number stays free text (from SAP).
    API: real-time search or periodic sync (TBD with Bookkeeping) · Current free text must be migrated
Why these are blockers, not just enhancements

The Bookkeeping integration (Phase 2) will generate structured accounting files per transaction. Those files must contain a valid legal entity and correct GL / cost center codes. If PR data is wrong, accounting files will be wrong — and fixing them retroactively is far more expensive than fixing the source.

1B — Vendor Linkage Enhancements (E1–E5)
  • E1
    Two vendor input modes at Price Comparison — Search & Select (vendor in master → locks vendor code + tax ID immediately) OR Manual Entry (vendor not yet in master → records name + tax ID as unconfirmed, flags as "Pending Vendor Registration")
    Field: mode toggle on each line item · UI already built — enforcement is the gap
  • E2
    Vendor identity frozen after CFO approval — winning vendor per line item cannot change after CFO approves. Vendor code, tax ID, and name are all locked.
    Lock applies to: vendor code, tax ID, vendor name, expected delivery date
  • E3
    Vendor match check at Waiting Create PO (manual entry path only) — Procurement searches vendor master by tax ID before document creation. Match on tax ID only (name variations acceptable). No match → register vendor first. Skip for master-selected vendors.
    Gate: blocks document creation until tax ID match confirmed · Only for Manual Entry path
  • E4
    Vendor field on document is read-only — pre-populated from confirmed vendor code at PO / Contract / Memo / Online Payment creation. Not editable.
  • E5
    Expected delivery date per winning vendor at Price Comparison — when Procurement selects the winning vendor per line item, they must also fill in the Expected Delivery Date. One date per line item. Mandatory before PR can advance past Price Comparison.
    Lock: frozen after CFO approval (same lock as E2) · Pre-populated on PO creation (read-only) · Visible on GR screen
Developer — Key Business Rules
Entity: required, defaults NTB, passed to all Bookkeeping files Cost center: one per PR (not per line item) GL code: per line item (Expense only) Asset number: free text, sourced from SAP Vendor lock: triggered on CFO approval event Tax ID match: exact match only, name is display only E5 date: per line item, not per PR, frozen with E2 lock Missing entity → blocks Bookkeeping file generation

Phase 3 — Remove SAP from the PO Step
PO Approval Process & Internal PO Number Generation
Generate and approve POs entirely inside this system — the SAP retirement milestone
6 / 9 After Phase 2
Business Objective
Today, every PO must go through SAP for a PO number and approval — Procurement exports data, creates it in SAP, then enters the number back. Phase 3 eliminates this round-trip. POs are created, numbered, and approved entirely within this system. SAP is no longer on the critical path.
This phase begins ONLY after Phase 2 exit criterion is met — the accounting backbone must be proven first.
What gets built in Phase 3
  • →
    Internal PO number generation
    Format: PO-{BE year}/{5-digit sequence} e.g. PO-2569/00001
    Assigned at Draft, never changes.
    Replaces manual SAP PO number entry · BE year = Gregorian + 543
  • →
    PO approval chain
    Financial Controller approves. CFO approves only when PO total exceeds configurable threshold.
  • →
    Rejection handling
    PO returns to Draft with reviewer comments. Procurement edits and resubmits.
  • →
    CFO threshold configuration
    CFO sets the amount threshold. Applies to new POs only. Stored as a config value.
    Field: configurable amount · Rule: applies to new POs from config date forward
  • →
    PO cancellation flow
    Procurement initiates with mandatory reason. Requires approval chain. Blocked if GR or confirmed Billing exists against this PO.
    Block condition: any confirmed GR or confirmed Billing → cancellation not allowed

SAP cutover decision
Deliberate Coordinated Cutover (not automatic)

Once Phase 3 is live and validated, the PO number field switches from manual SAP entry to auto-generated. This is a planned cutover event — in-flight SAP-numbered POs are NOT migrated. The cutover applies to new POs only from the cut date forward.

Developer — Key Business Rules
PO number: assigned at Draft, immutable Sequence: per BE year, resets at new year CFO gate: threshold is a config value, not hardcoded Cancellation blocked if GR confirmed OR Billing confirmed exists In-flight SAP POs: not migrated at cutover

PO Ceiling Rule
Cumulative billing cannot exceed PO value

Sum of all invoices (confirmed + pending) against a PO ≤ PO value.
Exception: cumulative total reaching PO value − 1 THB counts as fully billed (rounding tolerance).
Submission blocked if it would exceed PO value — error shown with remaining billable amount.

Vendor-Facing Invoice Statuses (3 only)
Waiting Review Confirmed Rejected
Why only 3 statuses?

Vendors do not see internal routing between Procurement and Accounting. Both "Waiting Procurement Review" and "Waiting Accounting Confirmation" map to "Waiting Review" from the vendor's perspective. Intentionally simple for non-technical vendor users.
